# 我给自己的播客频道做了一套 AI 内容生产工具——以及这个过程里学到的东西

这篇文章想做两件事：介绍这个工具，以及把做这个工具过程里踩过的坑和学到的东西写清楚。

这套工具叫 [kdb-video-pipeline](https://github.com/sunyuzheng/kdb-video-pipeline)，是我为自己的播客频道「课代表立正」从头搭的内容生产流水线。高度 customized——工具里内置了频道的 guideline、受众分析逻辑、过去播放量最高的 25 条标题作为参考基准，以及一套我自己摸索出来的高光片段选取原则。

你用起来会发现它确实能跑，但很多地方要生效需要你把类似的东西换成自己的。这不是通用工具，是一套思路和一个可以 fork 的起点。

---

## 这套工具做什么

从一个视频文件开始，自动完成六步：

1. **本地转录**：用 Qwen3-ASR（Apple Silicon 本地运行，完全离线）把语音转成 SRT 字幕
2. **字幕校对**：用 LLM（Claude / Gemini / OpenAI 任选）纠正专有名词、语气词、同音字
3. **断句**：把长句拆成每条 ≤20 字，适合导入剪辑软件
4. **生成文章**：提炼成频道风格的文章，归档用
5. **高光分析**：识别视频开头的高光片段，分析叙事弧线和受众心理
6. **标题生成**：三轮工作流，最终产出 6-10 个候选标题 + 封面建议

一个命令跑完：

```bash
venv/bin/python tools/process_video.py 视频.mp4 --seeds 嘉宾名
```

---

## 为什么要做这个

做播客的人大概都经历过这个：你录了一期不错的内容，但转录出来的字幕错字连篇，嘉宾名被识别成同音字，要么手动一条条改，要么就这么发出去，心里不痛快。

我录了很多期，攒下了 689 个历史视频。字幕质量参差不齐，标题靠感觉，高光片段靠剪辑师的直觉。没有一套可以复用的东西。

做这个工具的动机很简单：**让每一步都有人帮我想一遍，但最终决定还是我来拍。**

## 这个过程里学到了什么

### 1. 在哪里出问题，就在哪里解决，别指望下游兜底

最开始的思路是：Qwen 转录出来之后，交给 LLM 去纠错。于是我做了一个很复杂的六层校对流程——全局上下文分析、风险打分、滑窗校对、二次审核……

测出来的结果让我很意外：这个复杂系统的 CER 改善，还不如一个保守的候选词驱动策略的三分之一。

原因其实很简单：LLM 被允许自由改写，就会改出错（把对的改错）。而候选词驱动的策略只对有证据支持的 flag 表态，做不到的时候干脆不改——坏情况几乎全可追溯。

更重要的洞见是：**字幕出错的根本原因不在校对环节，在转录环节。** 第一次出现的嘉宾名，Qwen 没见过，会写成同音字，事后再纠错是亡羊补牢。正确做法是在转录前把嘉宾名注入 ASR 引导解码（`--seeds` 参数）。这个改动的效果，比任何校对算法都直接。

**在哪里出问题，就在哪里解决。别指望用下游的复杂度掩盖上游的信息缺失。**

### 2. 规则杀死创造力；认知开放可能性

标题生成最初的 prompt 是这样的：

> 生成 ≥25 个候选标题，覆盖以下 6 个入口角度：反常识型、场景代入型、身份认同型、框架升级型……

我以为这是在帮模型，实际上是在约束它。生成出来的标题像是在填格子：每个角度一个，整整齐齐，但都不够准。

后来我把这些全删掉，换成一句话：「角度跟着内容走，这期内容适合几个强入口就探几个，不要为了多样性凑数。」

生成质量明显好了。不是因为模型变聪明了，而是因为我不再要求它在特定格子里思考。

但问题来了：**删掉规则的同时，我也把一些有价值的认知一起删掉了。** 比如「高光组合要有内在叙事逻辑，不只是把戏剧性时刻堆在一起」——这不是规则，是判断力。

区别在哪里？规则是「必须按这个格式来」，认知是「有个思考维度你可能没想到」。前者限制模型，后者扩展模型。

最后的解法是：把认知沉淀到 guideline 文件里，让高光提取和标题生成都来读这个文件作为参考上下文。Prompt 里不再有任何硬性格式要求，只有角色和目标。

**用认知引导，用规则约束——这两件事的边界很重要。**

### 3. 高光是标题的地基，不是标题的复读机

这是整套工具里我最花时间想的部分。

标题和高光是两件不同的事：
- **高光**的功能是：让已经点进来的观众感觉「这期有料」，让他们不想关掉
- **标题**的功能是：让还没点进来的人决定「这期值得点进去」

两件事的对象不同，时机不同，逻辑也不同。

如果标题直接描述高光说了什么，等于剧透——点进来的人会觉得已经知道了，反而少了一层悬念。好的标题应该描述高光背后的那个更大的问题，让人带着好奇心进来，然后被高光「证实」这个好奇心是值得的。

这个模型帮我把一件模糊的感觉说清楚了：**标题创造期待，高光证实期待值得。两者是合谋关系，不是重复关系。**

### 4. AI 选的高光，不如编辑自己选的

工具里有一个机制：如果你在 SRT 文件末尾手动追加了高光字幕（时间戳会重置为 00:00:xx），系统会优先用这些亲选的片段做分析，不走 AI 全文扫描。

为什么要这么设计？因为 AI 扫描有一个固有倾向：**选戏剧性最高的时刻，而不是叙事上最关键的时刻。** 这两件事经常不一样。最吸引人的那句话，可能不是整期内容最值得做高光的切入点。

编辑自己选的高光，包含了他对这期内容的整体判断——「这几段放在一起，讲了一个更大的故事」。这个判断力，目前 AI 还学不会。

所以工具的设计原则是：**AI 处理执行层，人负责判断层。** 当编辑已经做了判断（手动选了高光），AI 负责把判断说清楚；当编辑没有判断，AI 提供一个候选，人再来验收。

### 5. 一个 prompt 再强，也抵不过一个流程

这是这套工具里最不寻常的设计，值得单独说清楚。

大多数人用 AI 生成标题的方式是：把内容粘进去，写一个 prompt，让 AI 出 10 条候选，挑一个改改用。

我们的做法不是这样。标题生成会依次调用 Claude Code 三次，每次都是一个完整的对话，整个过程可能花 10 分钟。

为什么要这样做？

三轮的设计不是为了「问三遍希望答案变好」，而是因为生成标题这件事，本质上包含三个**互相干扰**的认知过程——如果放在一个 prompt 里，它们会互相拉扯：

- **发散**：尽可能覆盖所有可能的切入角度，不要过早收敛
- **评估**：独立判断哪些方向有潜力，找到盲区，这个评审者不应该看到「自己刚刚生成的东西」
- **收敛**：带着前两轮的认知，做出最终决定

把这三个过程分开，每一步都能专注做好自己的事。Round 0 不用担心「候选太多」，Round 1 不会对自己的产出过度宽容，Round 2 拿到的是两轮完整的思考材料。

这是我在一次 AI builders 的分享活动上说过的一个观点：**做 AI 应用，很多人在优化 prompt，但真正的杠杆在流程设计。** 把一个创意任务拆成生成→评审→收敛三个阶段，用 pipeline 串起来，比把所有要求塞进一个 prompt 强得多。

标题生成这件事，不是一个创意问题，是一个工程问题。

### 6. Evaluation first：我说了，但没有真正做到

这是一个需要诚实面对的学习。

我在那次分享里说了 evaluation first 的重要性。这个项目里也有 evaluation 的痕迹——频道过去播放量最高的 25 条真实标题，作为 Round 1 评审的外部基准。让模型对比真实数据，比让它自我评价诚实得多。

但这只是 evaluation 的入门版本。我放进去的只是「好标题的列表」，没有告诉系统**为什么这些标题好**。它能看出「这些高播标题的风格」，却无法真正学到「这条标题好在哪里、那条标题差在哪里」背后的判断逻辑。

高光的 evaluation 更弱。目前 AI 扫描选出来的高光，我会看一眼觉得「差不多对」或「这期角度选错了」，但这个判断留在我脑子里，没有反哺系统。

**能省时间的工具和能持续提高质量的系统，差的就是这一层。**

现在这套工具能省掉我大量重复劳动，这已经有价值了。但如果我要真正提高输出质量，而不只是提高效率，下一步必须在 evaluation 上下功夫：收集真实的好标题和好高光，标注为什么好，把这些判断作为有标签的 ground truth 喂进系统。

这件事我还没做，但它是这个项目最重要的未竟之事。

## 一些实用细节

**跑长视频别让 Mac 睡着：**

```bash
caffeinate -i venv/bin/python tools/process_video.py 视频.mp4 --seeds 嘉宾名
```

**步骤 5-6 需要 Claude Code CLI 已登录**，如果卡住了，先跑 `claude -p "test"` 确认是否有效。

**高光选错了怎么办**：在 `.final.srt` 末尾追加你想要的高光字幕，重新跑 `generate_highlights.py`，系统会自动优先用你选的内容。

## 欢迎来用，但要做好 fork 的准备

这套工具是为我自己搭的，但搭的过程里的思路和架构是通用的：六步流水线、候选词驱动校对、高光驱动标题、认知 vs 规则的分层……这些在其他内容生产场景里应该也有用。

如果你想把它用在自己的频道上，最关键的三件事需要替换成你自己的：
1. `data/guideline_kedaibiao.md`：改成你的频道定位和受众描述
2. `data/top_titles.txt`：改成你的频道历史高播标题（这是 Round 1 评审的基准）
3. `data/channel_vocab.json`：改成你的频道词汇表（人名、品牌名、专有术语）

其余的逻辑应该可以直接用。
